> 자세한 부트스트랩 절차는 프로젝트 루트 `HANDOFF.md` 참고.

**프로젝트**: 도서관 메타포를 적용한 북마크 관리 웹 (포트폴리오 프로젝트, 작업 경로에 "포폴" 포함)

**문제 정의**: 구글 기본 북마크는 일렬 나열식이라 많아질수록 원하는 항목 찾기가 어려움.

**컨셉**: 북마크를 책장에 꽂힌 책처럼 시각화한 도서관 인터페이스.

**핵심 UX 흐름**:
- 첫 로그인 → 도서관 전체 풍경 노출
- 각 책장 = 북마크 폴더 (단, 한 칸당 최대 권수 제한 있음 — 도서관 메타포 유지를 위해 의도적으로 둠)
- 책장 클릭 → 줌인 → 꽂힌 책들과 각 책의 제목(사이트명 또는 사용자 지정) 표시
- 표지 색상 대비 자동 처리(흰색/검정 기본), 사용자 오버라이드 허용
- 즐겨찾기 → "베스트셀러" 탭에 상위 10~20권만 노출
- 북마크가 늘수록 도서관에 실루엣 인구 증가 (게이미피케이션, 자세한 묘사는 피하고 실루엣 수준)

**필수 기능**:
- 드래그 앤 드롭으로 책 위치 이동
- 검색 (북마크 양이 많아질 것을 전제하므로 필수)
- 도서관/책장 색상·분위기 커스터마이징 (그 자체가 콘텐츠)

**북극성(North Star)**: 사용자 본인의 발화 — "누군가가 내가 만든 북마크 도서관 UI를 꾸며서 공유해 올리는 그런것."
즉, 이 제품의 성공 지표는 **사용자 수**나 **북마크 수**가 아니라 **꾸며서 자랑하고 싶어지는가**임. Notion/Pinterest/Tumblr/싸이월드 미니홈피 계보의 "도구가 아니라 캔버스"형 제품.

**Why**: 단순 도구가 아니라 "잘 정리된 책장을 보면 뿌듯함을 느끼는" 정서적 보상까지 노리는 제품. 사용자가 자기를 투영하는 캔버스가 되는 것이 목표. 따라서 시각/인터랙션 품질이 기능 정확성만큼 중요.

**v1.0 종료 조건 가설**: "내 도서관을 트위터/디스코드에 링크로 공유했을 때, 친구가 미리보기 이미지에 끌려 클릭하고 가입 후 자기 도서관도 꾸미기 시작한다." 입소문 1대1 전파가 자연스럽게 일어나는 시점.

**v1.0 필수**: 로그인 + 공개 프로필 URL(`/u/{username}`), 도서관 시각화, 북마크 직접 추가/편집/드래그, 다차원 커스터마이징(팔레트/책장 스타일/표지 색), 공유 URL + 자동 OG 이미지, 공개/비공개 토글(책 또는 책장 단위), PostgreSQL.

**v1.0 후순위 (의도적 제외)**: Chrome 확장(직접 동기화), 모바일 동등 지원, 갤러리/피드, 테마 마켓플레이스, 결제, 베스트셀러 탭 별도 진열.

**Chrome 북마크 import는 v1.0에서 구현됨** (2026-04-30): HTML 파일 업로드 방식으로, "자동으로 책장 만들기" 또는 "창고로 모두" 두 가지 모드 제공. 별도 확장 없이 동작.

**비주얼 디자인 방향 (확정)**:
- 사이트 UI: 라이트 크림 톤 (`#FAF7F2` 배경, `#2A2520` 텍스트, `#8B5A3C` 월넛 액센트) — 갤러리 벽 컨셉, 책장이 주인공이 됨
- 책장 테마 3종: **Dark Walnut(기본)** / Warm Pine(밝은 메이플) / Industrial Frame(스틸+우드)
- 컨셉 레퍼런스: 한국/북유럽 카페 도서관, 메이플 격자 책장에 책+화병/도자기/식물 섞여있는 무드
- 폰트: Pretendard 등 한글 sans-serif 표준, 제목은 약간의 명조계 검토 가능

**미래 수익화 훅 (지금 코드만 준비, 구현 X)**:
- BookshelfTheme 데이터 모델은 처음부터 `tier` (FREE/PAID) + `price` 필드 가짐
- v1.0에서는 모든 테마 `tier=FREE` — 결제 로직/약관/PG 연동은 v2.0 이후
- 이 구조 덕분에 미래 전환 시 테이블 추가 + 체크 로직만 얹으면 됨
- 추후 후보 BM: 데코레이션 아이템(화병/식물), 도서관 배경(시간대/날씨), 특수효과

**v2.0 다중 도서관 (확정된 미래 방향)**:
- 한 사용자가 여러 도서관 보유 가능 (사용자 발화: "공유한 도서관 때문에 새로 만들고 싶은 도서관이 없어지면 안 된다")
- 데이터 모델 변경: `User → Library` 1:1 → 1:N
- URL: `/u/{username}/{librarySlug}`, 각 도서관 독립 공개/비공개
- 단, **창고(Storage)는 사용자 단위 1:1 유지** — 도서관이 늘어도 창고는 하나
- 현재 v1.0 구현 시 이 확장을 닫지 않도록 주의 (필드 이름/관계 설계)

**창고(Storage) 컨셉**:
- 정체성: "도서관"은 큐레이션된 공간 (책장당 30권 의도된 제약). "기록보관소"가 아님.
- 수천 권 import 사용자를 위한 안전판: 별도 Storage 엔티티 + StoredBook (User 직속, Library 무관)
- StoredBook은 꾸밈 정보 없음(URL/제목/원래 폴더 경로만), 책장으로 옮길 때 Book으로 변환
- 창고는 항상 비공개, 평면도 시각화 시 입구 옆 박스 더미로 표현 (v1은 단순 리스트)
- import 흐름: 사용자가 "자동으로 책장 만들기" 또는 "창고로 모두" 둘 중 선택 (UI에서 둘 다 명시 표시 필수)

**Chrome 북마크 import 정책**:
- HTML 파일 업로드 (Chrome 내보내기 형식, Netscape Bookmark)
- 폴더 깊이 무제한 입력을 평탄화: **책이 직접 든 폴더만 책장이 됨**, 컨테이너 폴더는 무시
- 이름 충돌 시 부모 경로 자동 추가 (`자료 / 기타`)
- 30권 초과 폴더는 자동 분할 (`이름 (1)`, `이름 (2)`) 또는 사용자가 창고 선택 가능
- 중복 URL 감지는 v1에선 안 함 (사용자가 수동 정리)

**전략적 포지셔닝**: 포트폴리오 → 씨앗 서비스 트랙. v1.0은 포트폴리오 명작으로 완결되도록 하되, 코드/데이터 모델은 차후 확장 가능하게 설계 (REST API 우선, PostgreSQL부터 사용, 인증 토큰 기반).

**How to apply**:
- 기능 추가/설계 제안 시 두 질문을 통과해야 함: (1) 도서관 메타포에 부합하는가 (2) 사용자가 꾸미고 자랑하고 싶어지는 데 기여하는가
- "효율적 도구" 관점의 단순화 제안(목록형 fallback, 책장 용량 제한 제거 등)은 북극성과 충돌 — 효율성 이유로 컨셉 후퇴 금지
- 로그인 후 첫 화면은 "도서관 전체 풍경"이 되어야 함 — 추후 라우팅/`defaultSuccessUrl` 설계 시 반영
- OG 이미지 자동 생성은 이 제품의 "마케팅 그 자체" — v1.0 내에서 반드시 다룸 (자리만 비워두고 나중에 구현해도 됨, 그러나 출시 전엔 필수)
